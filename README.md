## Overview

**First Line of Defense** is a lightweight, dependency-free Bash script designed to give students and sysadmins a quick, structured snapshot of a Linux system's security posture. It runs three audit phases — System Information, Threat Detection, and Hardening Checks — and writes all findings to `security_report.txt`.

The project was built to bridge the gap between *why* we audit (global security standards) and *how* we audit (practical Bash scripting).

---

## Features

| Category | What It Checks |
|---|---|
|  System Info | OS version, kernel, architecture, hostname |
|  Identity | Current user, least-privilege enforcement |
|  Patching | Upgradable packages via `apt` / `yum` / `dnf` |
|  Threat Detection | Suspicious file extensions in `/tmp` (`.exe`, `.bat`, `.ps1`, `.vbs`) |
|  Malware Indicators | World-writable `.sh` scripts, dummy malware file check with SHA256 |
|  Firewall | `ufw`, `firewalld`, and `iptables` detection and status |
|  SSH Hardening | `PermitRootLogin` and `PasswordAuthentication` settings |
|  Sudo Audit | All members of the `sudo` group + `/etc/sudoers` entries |
|  Password Policy | `PASS_MAX_DAYS`, `PASS_MIN_DAYS`, `PASS_WARN_AGE` from `/etc/login.defs` |
|  Reporting | Colour-coded terminal output + clean `security_report.txt` |

## Requirements

- A **Linux** system (tested on Ubuntu 24.04 LTS)
- **Bash** 4.0 or later (`bash --version`)
- No external dependencies — uses only standard GNU/Linux utilities
- Works on:
  - Ubuntu / Debian (uses `apt`)
  - RHEL / CentOS / Fedora (uses `yum` / `dnf`)
  - Windows Subsystem for Linux (WSL 1 & 2)
  - Any Linux VM

## Installation & Usage

**1. Clone the repository**

```bash
git clone https://github.com/your-username/Defense-Audit-Tool.git
cd Defense-Audit-Tool
```

**2. Make the script executable**

```bash
chmod +x Defense-audit-tool.sh
```

**3. Run the audit**

```bash
# Run with sudo for full access to all checks
sudo bash Defense-audit-tool.sh
```

**4. View the report**

```bash
cat security_report.txt
```

> **Tip — Lab Setup:** To simulate a threat detection hit before running the script, create the dummy malware file first:
> ```bash
> touch /tmp/totally_not_malware.sh
> chmod 777 /tmp/totally_not_malware.sh
> sudo bash first_line_of_defense_audit.sh
> ```

## Audit Phases Explained

### Phase 1 — System Information & Standards
Establishes an asset and identity baseline before any protective controls are evaluated. Checks the OS version, the privilege level of the session, and whether all packages are patched — the most fundamental hygiene requirement in any security framework.

### Phase 2 — Threat Detection Simulation
Simulates the work of a lightweight Endpoint Detection & Response (EDR) tool. Scans `/tmp` for file extensions commonly associated with Windows malware dropped onto Linux systems (`.exe`, `.bat`, `.ps1`, `.vbs`) and flags shell scripts with dangerous world-writable permissions. Also checks for a specific dummy malware indicator file used in lab environments.

### Phase 3 — Linux Hardening
Evaluates the system against common CIS Benchmark and ISO 27001 hardening controls: firewall status, SSH configuration, over-privileged accounts, and password aging policy. These are the most frequently misconfigured settings on out-of-the-box Linux installations.

## NIST / ISO Control Mapping

| Check | NIST CSF v2 | ISO/IEC 27001:2022 |
|---|---|---|
| OS identification | ID.AM – Asset Management | A.8.1 Inventory of Assets |
| Least privilege check | GV.RR – Roles & Responsibilities | A.8.2 Privileged Access Rights |
| Patch status | ID.RA / PR.PS – Vuln & Config Mgmt | A.8.8 Technical Vulnerability Mgmt |
| Suspicious files in /tmp | DE.AE – Anomalies & Events | A.8.7 Protection Against Malware |
| Dummy malware indicator | DE.CM – Continuous Monitoring | A.8.7 / A.5.7 Threat Intelligence |
| World-writable files | PR.DS – Data Security | A.8.3 Information Access Restriction |
| Firewall status | PR.AA / PR.IR – Network Security | A.8.20 Network Security Controls |
| SSH hardening | PR.AA – Access Control | A.8.20 / A.8.2 Privileged Access |
| Sudo privilege review | GV.RR – Roles & Responsibilities | A.8.2 Privileged Access Rights |
| Password policy | PR.AA-01 – Identity Management | A.5.17 Authentication Information |

For a full one-page breakdown, see [`nist_iso_summary.pdf`](./nist_iso_summary.pdf).

## Exit Codes

The script exits with a code equal to the number of `[FAIL]` findings. This makes it easy to integrate into automated pipelines.

| Exit Code | Meaning |
|---|---|
| `0` | All checks passed — no critical failures |
| `1` | 1 critical failure detected |
| `N` | N critical failures detected |

**Example — fail a CI job if any critical issue is found:**
```bash
sudo bash first_line_of_defense_audit.sh || { echo "AUDIT FAILED"; exit 1; }
```

## Extending the Script

The script is intentionally simple — here are some ideas for taking it further:

- **Auto-remediation:** Add a `-fix` flag that automatically enables `ufw`, sets SSH options, or removes world-writable permissions when issues are found.
- **Email alerts:** Pipe `security_report.txt` to `mail` or a webhook (Slack, Teams) after each run.
- **Cron scheduling:** Run the audit nightly and diff reports to catch changes.
- **Custom IOC list:** Replace the single dummy malware filename with a loop over a configurable list of known indicator filenames.
- **JSON output:** Add a `-j` flag to emit results as JSON for ingestion into a SIEM or log aggregator.
